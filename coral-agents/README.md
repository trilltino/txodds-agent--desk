# coral-agents

CoralOS participant stubs for the TxODDS Agent Desk.

## Architecture: one orchestrator, track-specialist handoffs

The desktop app runs a **multi-agent graph on the local Coral bus**. There is
one *orchestrating* agent (`match-intelligence-agent`) and one
*specialist* agent per track, promoted by a `DELEGATE` `TOOL_CALL` at the
end of each run.

All message publishing currently happens in the Tauri Rust backend
(`native/src/services/agent/runtime.rs`) via the puppet API — the Python
stubs here provide the Docker images CoralOS needs to register each
participant. However the participant slots, Coral message protocol, and
`DELEGATE` handoff are all fully wired for each specialist to become a
**real independent agent** with its own logic.

## How the handoff works (from `runtime.rs`)

```rust
// --- multi-agent handoff: delegate to the track specialist ---
let specialist = specialist_for_track(context.track);
emit_message(... MATCH_INTELLIGENCE_AGENT → specialist, CoralVerb::ToolCall, "DELEGATE ...");
emit_message(... specialist → [MATCH_INTELLIGENCE_AGENT, USER_PROXY], CoralVerb::ToolResult, ack);
```

`specialist_for_track` maps:

| TrackMode | specialist participant |
|---|---|
| `Settlement` | `settlement-agent` |
| `Trading` | `sharp-movement-detector` |
| `Fan` | `fan-pundit-agent` |

The specialist acknowledgement text is today generated by `specialist_ack()`
in Rust. Promoting a slot to a real independent agent means replacing that
scripted ack with the agent's own MCP message — no protocol changes required.

## Participant directories

| Directory | Docker image | Status | What publishes its messages today |
|---|---|---|---|
| `match-intelligence-agent` | `match-intelligence-agent:0.1.0` | **Orchestrator** | Rust runtime (full pipeline logic) |
| `proof-guard-agent` | `proof-guard-agent:0.1.0` | Gate | Rust `services::proof` — deterministic txoracle gate |
| `settlement-agent` | `settlement-agent:0.1.0` | Specialist (Settlement track) | Rust `services::chain` — scripted ack; ready for own logic |
| `sharp-movement-detector` | `sharp-movement-detector:0.1.0` | Specialist (Trading track) | Rust `services::agent` — scripted ack; ready for own logic |
| `fan-pundit-agent` | `fan-pundit-agent:0.1.0` | Specialist (Fan track) | Rust `services::llm` — scripted ack; ready for own logic |

Each directory contains: `agent.py` (idle MCP client), `coral-agent.toml`,
`Dockerfile`, `requirements.txt`, `README.md`.

## Why the extra stubs exist

`protocol.rs` declares `ALL_AGENTS` with all nine participant names.
`console.rs` registers each in the CoralOS session with `"runtime": "docker"`.
CoralOS requires a resolvable Docker image for every registered participant.
The Python stubs supply that image — a minimal MCP client that connects and
idles — until each specialist is promoted to independent logic.

## Full Coral Console message flow

Every run emits this transcript. Messages in `()` are published by the Rust
runtime as that participant via the puppet API:

```
round 1  (txline-ingest)           → match-intelligence-agent   OBSERVED
round 2  (txline-normalizer)       → match-intelligence-agent   NORMALIZED
round 3  (user-proxy)              → match-intelligence-agent   WANT
round 4  match-intelligence-agent  → (feature-extractor)        TOOL_CALL
round 4  (feature-extractor)       → match-intelligence-agent   TOOL_RESULT
round 5  match-intelligence-agent  → (proof-guard-agent)        PROOF_REQUESTED
round 5  (proof-guard-agent)       → match-intelligence-agent   PROOF_RECEIVED
round 5  (proof-guard-agent)       → match-intelligence-agent   VALIDATION_SIMULATED
round 6  match-intelligence-agent  → (user-proxy)               AGENT_THOUGHT (Venice LLM)
round 7  match-intelligence-agent  → (user-proxy)               SIGNAL
round 7  match-intelligence-agent  → (user-proxy)               TOOL_RESULT (action)
round 8  match-intelligence-agent  → <specialist>               TOOL_CALL  ← DELEGATE
round 8  <specialist>              → match-intelligence-agent   TOOL_RESULT (ack)
           + user-proxy
round 9  match-intelligence-agent  → (user-proxy)               EVALUATED
```

`<specialist>` = `settlement-agent` | `sharp-movement-detector` | `fan-pundit-agent`
depending on `TrackMode`.

## Policy guardrails

All hard limits live in the Rust policy layer:

- No participant can authorise a payment (`allow_money_decisions = false`)
- On-chain release requires explicit wallet user approval (`allow_settlement_release = false`)
- LLM output is advisory only — Venice explanations cannot flip the proof gate

`proof-guard-agent` alone has `allow_proof_verdicts = true`; the txoracle
gate is deterministic Rust, not an LLM.
